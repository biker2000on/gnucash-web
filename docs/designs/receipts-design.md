## Problem Statement

GnuCash Web is a read-only PWA for viewing GnuCash financial data. Users currently have no way to attach receipts to transactions, leading to disconnected financial records. During tax prep or audits, users (and their accountants/partners) must manually cross-reference receipts stored elsewhere with transactions in the ledger. The goal is to bring paperless-ngx-quality document management directly into the transaction workflow — capture, store, view, OCR search — all linked to transactions.

## What Makes This Cool

A self-hosted GnuCash web viewer that also manages receipts eliminates an entire category of financial paperwork friction. Snap a receipt on your phone, it's instantly linked to the transaction, OCR-indexed, and searchable. Your accountant opens the ledger, clicks a paperclip icon, and sees the receipt. No separate app, no folder digging, no "where did I put that receipt?"

## Constraints

- Must work on mobile (PWA) with native camera capture
- Must work on desktop with drag-and-drop
- Storage must be local/self-hosted (no cloud dependencies)
- Must support JPEG, PNG, and PDF formats
- OCR is a must-have for receipt text search
- Consistent with existing GnuCash Web UI patterns (Tailwind, Modal component, React Query)
- Read-write feature in an otherwise read-only app — receipts are app-managed data, not GnuCash data

## Premises

1. **Receipts must be linked to transactions** — a standalone document store isn't enough; the value is the transaction-to-receipt link
2. **Both capture and viewing must be excellent** — mobile camera capture needs to be frictionless; the view experience needs to serve an accountant pulling up receipts during tax prep
3. **OCR is a must-have** — searching across hundreds of receipts by text is how accountants actually work
4. **Storage should be local/self-hosted** — no cloud dependencies, consistent with the GnuCash self-hosting ethos

## Approaches Considered

### Approach A: Paperless-Lite (Minimal Viable)

Filesystem storage with a `receipts` DB table. Upload from the transaction detail view. Simple modal viewer (image or embedded PDF). OCR via Tesseract in Docker or installed binary. Search via SQL `ILIKE` on extracted text.

- **Effort:** M | **Risk:** Low
- **Pros:** Ships fast, no new infra, proven patterns from import/export code
- **Cons:** No thumbnails, PDF viewing depends on browser, no batch upload
- **Reuses:** Modal component, FormData upload pattern, Prisma schema
- **Completeness:** 6/10

### Approach B: Full Paperless Experience (Ideal Architecture)

Everything in A, plus: auto-generated thumbnails (sharp/canvas), a dedicated `/receipts` page with gallery view and full-text search, batch upload with progress, receipt preview thumbnails in transaction ledger rows (paperclip icon with hover preview).

- **Effort:** L | **Risk:** Med
- **Pros:** Closest to paperless-ngx quality, delightful for both uploader and viewer
- **Cons:** More moving parts (image processing, thumbnail generation), larger scope
- **Reuses:** Everything from A + IntersectionObserver pattern for gallery infinite scroll
- **Completeness:** 9/10

### Approach C: Receipt-First Workflow (Creative/Lateral)

Flip the flow: upload receipts first (bulk), OCR extracts amount/date/vendor, then auto-suggest which transaction to link to. Unmatched receipts sit in an inbox until linked.

- **Effort:** L | **Risk:** Med-High
- **Pros:** Most natural mobile workflow (snap now, link later), closest to Expensify UX
- **Cons:** Auto-matching is hard to get right, requires good OCR, more complex state management
- **Reuses:** Everything from A + new matching algorithm
- **Completeness:** 8/10

## Recommended Approach

**Approach B: Full Paperless Experience** — chosen because it delivers the paperless-ngx quality the user described while staying grounded in patterns the codebase already uses. The thumbnail generation and gallery page are additive complexity with low architectural risk.

### Database Schema

New Prisma model:

```prisma
model GnucashWebReceipt {
  id               Int      @id @default(autoincrement())
  book_guid        String   @db.VarChar(32)  // scoped to the active book
  transaction_guid String?  @db.VarChar(32)  // nullable for unlinked receipts (upload now, link later)
  filename         String   @db.VarChar(255)
  storage_path     String   @db.VarChar(500)
  thumbnail_path   String?  @db.VarChar(500)
  mime_type        String   @db.VarChar(100)
  file_size        Int
  ocr_text         String?  @db.Text
  ocr_status       String   @default("pending") @db.VarChar(20) // pending, processing, complete, failed
  created_at       DateTime @default(now())
  updated_at       DateTime @default(now())
  created_by       Int      // REFERENCES gnucash_web_users(id) ON DELETE CASCADE

  @@map("gnucash_web_receipts")
  @@index([transaction_guid])
  @@index([book_guid])
}
```

**Referential integrity note:** Like all other `gnucash_web_*` tables, this table does not have a foreign key constraint to the GnuCash `transactions` table. Integrity is application-enforced. If a transaction GUID is invalid, the upload API returns a 404. If a transaction is deleted from GnuCash, orphaned receipts remain viewable from the `/receipts` page and can be re-linked or deleted. All queries are scoped by `book_guid` to prevent cross-book data leakage.

**Authorization:** All authenticated users can upload, view, and search receipts. Only the uploading user or an admin can delete a receipt. Uses the existing `requireRole()` pattern.

**Upload validation:** MIME type verified via magic bytes (not just file extension). Filenames sanitized to prevent path traversal. Allowlisted extensions: `.jpg`, `.jpeg`, `.png`, `.pdf`. Max file size: 10MB per receipt.

**Migration strategy:** Uses the `db-init.ts` pattern with `CREATE TABLE IF NOT EXISTS gnucash_web_receipts (...)` raw DDL, consistent with all other `gnucash_web_*` tables. The Prisma schema is updated for type generation only.

**Runtime query pattern:** Receipts use raw SQL via `db.ts` (`pool.query()`). Most `gnucash_web_*` entity CRUD uses Prisma Client, but raw SQL is chosen here specifically because receipt listing/search queries JOIN against GnuCash core tables (transactions, accounts, splits) that are only queried via raw SQL. Using raw SQL avoids mixing query patterns within the same request.

### Storage Architecture

Two storage backends, selected via `RECEIPT_STORAGE` env var:

**1. Filesystem (default):**
- `RECEIPT_STORAGE=filesystem` (or unset)
- `RECEIPTS_DIR` env var, defaults to `./data/receipts/`
- **Receipt files:** `{RECEIPTS_DIR}/{yyyy}/{mm}/{uuid}.{ext}` (keyed by UUID, not transaction — avoids file moves when linking/unlinking)
- **Thumbnails:** `{RECEIPTS_DIR}/{yyyy}/{mm}/{uuid}_thumb.jpg`
- **Backup note:** `RECEIPTS_DIR` must be included in backups alongside the database. Orphaned DB records (file missing) show a "file not found" placeholder in the UI. Orphaned files (DB record missing) are harmless and can be cleaned up manually.

**2. S3/MinIO-compatible object storage:**
- `RECEIPT_STORAGE=s3`
- `RECEIPT_S3_ENDPOINT` — MinIO/S3 endpoint URL (e.g., `http://minio:9000`)
- `RECEIPT_S3_BUCKET` — bucket name (default: `gnucash-receipts`)
- `RECEIPT_S3_ACCESS_KEY` / `RECEIPT_S3_SECRET_KEY` — credentials
- `RECEIPT_S3_REGION` — region (default: `us-east-1`)
- **Object keys:** `{yyyy}/{mm}/{uuid}.{ext}` and `{yyyy}/{mm}/{uuid}_thumb.jpg`
- Uses `@aws-sdk/client-s3` (new dependency) — compatible with MinIO, AWS S3, DigitalOcean Spaces, Backblaze B2
- Presigned URLs for serving files (avoids proxying large files through Next.js)

**Storage abstraction:** A `StorageBackend` interface with `put(key, buffer)`, `get(key)`, `delete(key)`, `getUrl(key)` methods. Two implementations: `FilesystemStorage` and `S3Storage`. The API routes use the interface — backend is selected once at startup based on `RECEIPT_STORAGE` env var.

**Shared:**
- Thumbnail generation via `sharp` (new dependency — must be added to `package.json`; Next.js may bundle it for image optimization but it's not guaranteed to be installed)
- **PDF thumbnails:** `sharp` cannot render PDFs directly. Use `pdfjs-dist` to render the first page to a canvas/buffer, then pass to `sharp` for thumbnail generation. This is a required additional dependency.

### API Routes

| Method | Route | Purpose |
|--------|-------|---------|
| POST | `/api/receipts/upload` | Upload receipt(s) with transaction_guid |
| GET | `/api/receipts/[id]` | Serve receipt file |
| GET | `/api/receipts/[id]/thumbnail` | Serve thumbnail |
| DELETE | `/api/receipts/[id]` | Delete receipt + files |
| GET | `/api/receipts` | List/search receipts (pagination, OCR text search) |
| GET | `/api/transactions/[guid]/receipts` | Get receipts for a transaction |

### OCR Pipeline

1. Upload receipt → save file → create DB record with `ocr_status: "pending"` → return success response
2. **Async OCR mechanism:** After the response is sent, use `after()` from `next/server` (stable in Next.js 15+) to run OCR after the response stream closes. This guarantees the OCR work completes even in serverless environments. Fallback for environments where `after()` is unavailable: fire-and-forget `Promise` with error logging (acceptable for self-hosted Docker deployments where the process is long-lived).
   - For images: run Tesseract on the image
   - For PDFs: extract text directly via `pdfjs-dist` (`getDocument().getPage().getTextContent()`), fall back to Tesseract on rendered pages if no text layer exists
3. Update `ocr_text` and `ocr_status` in DB. On failure, set `ocr_status: "failed"` and log the error. No automatic retry — user can trigger re-OCR from the receipt view modal.
4. **Search:** `WHERE ocr_text ILIKE '%query%'` for initial implementation. At ~1000+ receipts, migrate to PostgreSQL full-text search with a `tsvector` column and GIN index for sub-second search.

**Tesseract integration:** Production (Docker) uses system `tesseract-ocr` binary via `node-tesseract-ocr` for speed (~1-3 seconds per image). Development uses `tesseract.js` (WASM, ~10-30 seconds, no install needed) as fallback. Auto-detected: if system `tesseract` binary exists, use it; otherwise fall back to `tesseract.js`.

**Runtime query pattern:** Receipt queries use raw SQL via `db.ts` (the `pool.query()` pattern), consistent with all other queries in the codebase. The Prisma schema is used for migrations and type generation only, not for runtime queries.

### UI Components

1. **ReceiptUploadModal** — drag-and-drop zone + camera button (mobile via `capture="environment"`) + file picker. Shows upload progress. Accepts multiple files.

2. **ReceiptViewModal** — displays receipt image (or embedded PDF via `<iframe>`). Buttons: "Open in New Tab" (links to `/api/receipts/[id]`), "Download", "Delete". Shows OCR text in collapsible section.

3. **ReceiptIndicator** — paperclip icon in transaction rows. Green 📎 if receipt exists, gray ○ if not. Click opens view or upload modal respectively. Hover shows thumbnail preview on desktop.

4. **ReceiptsPage** (`/receipts`) — search bar + thumbnail grid with infinite scroll. Click opens ReceiptViewModal. Filters: date range, linked/unlinked. **Phasing note:** The gallery page can ship as Phase 2 after the core upload/view/link/OCR flow (Steps 1-7 in Next Steps). The core experience (attach receipts to transactions, view them, search via OCR) is complete without it.

### Mobile Experience

- `<input type="file" accept="image/*,application/pdf" capture="environment">` triggers native camera
- Upload modal goes fullscreen on mobile (existing Modal behavior)
- Gallery page uses 2-column grid on mobile, 3+ on desktop
- Touch-friendly: large tap targets, swipe to close modal

### Multiple Receipts per Transaction

Support multiple receipts per transaction (e.g., itemized receipt + credit card statement). The view modal shows a carousel/list when multiple receipts exist.

## Open Questions

1. **PDF rendering in modal** — `<iframe>` works but is clunky on mobile. Could render first page as image for preview using `pdfjs-dist` (already a dependency for thumbnails). Worth the added complexity? Default: use `<iframe>` initially, upgrade to rendered preview if UX feedback warrants it.
2. **Batch upload from gallery** — Allow uploading multiple unlinked receipts from the `/receipts` page? Default: yes, since `transaction_guid` is nullable and supports unlinked receipts.

## Success Criteria

- User can snap a receipt on mobile and attach it to a transaction in under 10 seconds
- User can drag-and-drop a receipt on desktop and see it attached immediately
- Receipts are viewable in a modal with "Open in New Tab" button
- OCR text is searchable from the `/receipts` page
- Thumbnails appear in the transaction ledger rows
- An accountant can find a receipt for a specific transaction without help

## Engineering Review Decisions (2026-03-21)

Decisions made during `/plan-eng-review`:

1. **OCR pipeline: BullMQ** (not `after()`). BullMQ + Redis already installed. Supports retries, concurrency limits, survives restarts.
2. **Storage keys, not paths.** DB stores object keys (`2026/03/abc123.jpg`), not backend-specific paths. Backend-portable.
3. **Filesystem proxy accepted.** Filesystem backend proxies files through Next.js API route. Fine for self-hosted.
4. **PATCH /api/receipts/[id] added.** For linking/unlinking receipts to transactions after upload.
5. **Self-contained ReceiptIndicator.** But receives `receiptCount` as prop (bulk JOIN) to avoid N+1. Owns click-to-modal behavior.
6. **Single ReceiptModal with tabs.** View + Upload in one modal. Opens to Upload if empty, View if receipts exist.
7. **Full test suite (20 tests).** All codepaths covered including security validation.
8. **Bulk receipt counts via JOIN.** Transaction list queries include receipt_count. No per-row API calls.
9. **Failure mode fixes.** Catch BullMQ enqueue failure (set `enqueue_failed` status). Delete file before DB record.
