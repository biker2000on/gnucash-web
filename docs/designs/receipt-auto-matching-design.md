## Problem Statement

GnuCash Web has receipt attachment and OCR, but linking receipts to transactions is manual — you must navigate to a specific transaction and upload from there. There's no way to upload receipts in bulk and have them matched to transactions automatically. Additionally, receipt search uses `ILIKE` which degrades at scale.

Two deferred TODOs from the receipts feature:
1. **Auto-Matching** — "snap now, link later" workflow with fuzzy matching
2. **Full-Text Search** — migrate from `ILIKE` to `tsvector` + GIN index

## What Makes This Cool

Upload a pile of receipts (tax-prep shoebox mode), and the app reads each one, extracts the amount/date/vendor, and says "this looks like your $42.17 Costco charge from Tuesday — link it?" Process your receipt backlog like email. Works without any AI dependency, but optionally connects to Ollama/OpenAI for better extraction accuracy.

## Constraints

- Must work fully without any AI provider (regex-based extraction as default)
- AI provider is optional enhancement via pluggable interface (Ollama, OpenAI, any OpenAI-compatible API)
- Self-hosted — no cloud dependencies
- Amount is the strongest matching signal; vendor matching is the hardest
- Full-text search and auto-matching are independent features
- Consistent with existing GnuCash Web UI patterns (Tailwind, Modal, React Query)
- Receipt data extraction stored as structured JSON for future features (line items, tax reports)
- PostgreSQL 12+ required (for generated columns)

## Premises

1. **Amount is the strongest matching signal.** Two transactions on the same day for $42.17 at different stores is rare in personal finance. Amount alone gets you 80% of the way.
2. **AI is optional enhancement, not a requirement.** The scoring algorithm must work fully without any AI — regex-based extraction is good enough for most receipts.
3. **Full-text search and auto-matching are independent features.** They share the OCR text column but should be built as separate modules that ship independently.
4. **The inbox workflow IS the product.** The matching algorithm is table stakes — what makes this delightful is the flow: upload in bulk, review matches in a focused inbox session.

## Approaches Considered

### Approach A: Minimal Matching + Full-Text Search
Regex extraction, inline suggestions on gallery, no inbox, no AI.
- **Effort:** M | **Risk:** Low
- **Pros:** Ships fast, FTS immediately useful
- **Cons:** No inbox, no batch capture, regex breaks on messy OCR
- **Completeness:** 5/10

### Approach B: Full Inbox + Scoring + Pluggable AI (CHOSEN)
Dedicated inbox tab, batch capture mode, multi-signal scoring engine, pluggable AI provider, structured extraction data stored for future use.
- **Effort:** L | **Risk:** Med
- **Pros:** Complete inbox workflow, AI-enhanced but works without it, structured data enables future features
- **Cons:** More moving parts, needs AI config
- **Completeness:** 9/10

### Approach C: AI-First with Learning
All extraction through AI, learning loop from user corrections, correction history for training.
- **Effort:** XL | **Risk:** High
- **Pros:** Most accurate long-term
- **Cons:** Requires AI, learning loop is complex, cold start problem
- **Completeness:** 7/10

## Recommended Approach

**Approach B: Full Inbox + Scoring + Pluggable AI** — delivers the complete "snap now, link later" workflow with the inbox as the centerpiece. AI is optional (works on day one with regex), and the structured extraction data enables line items, tax reports, and smart categorization later.

## Database Schema Changes

All schema changes are applied via `db-init.ts` using `CREATE TABLE IF NOT EXISTS` and `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` guards, consistent with all existing extension tables.

### New column on `gnucash_web_receipts`:
```sql
ALTER TABLE gnucash_web_receipts
ADD COLUMN IF NOT EXISTS extracted_data JSONB;
```

The `extracted_data` JSONB column stores structured extraction results:
```json
{
  "amount": 42.17,
  "currency": "USD",
  "date": "2026-03-15",
  "vendor": "COSTCO WHOLESALE #482",
  "vendor_normalized": "costco",
  "extraction_method": "regex",
  "confidence": 0.85,
  "dismissed_guids": ["abc123"]
}
```

Field reference:
- `amount`, `currency`, `date`, `vendor`, `vendor_normalized` — extracted receipt data
- `extraction_method` — `"regex"`, `"ai"`, or `"ai_fallback_regex"` (AI configured but failed, fell back to regex)
- `confidence` — 0.0-1.0 extraction confidence
- `dismissed_guids` — array of transaction GUIDs the user dismissed as "not this one" (persists across page reloads)
- Future expansion: `line_items`, `tax`, `subtotal` (not populated in this phase)

### Full-text search column (generated):
```sql
ALTER TABLE gnucash_web_receipts
ADD COLUMN IF NOT EXISTS ocr_tsvector tsvector
  GENERATED ALWAYS AS (to_tsvector('english', COALESCE(ocr_text, ''))) STORED;

CREATE INDEX IF NOT EXISTS idx_receipts_ocr_fts
  ON gnucash_web_receipts USING GIN (ocr_tsvector);
```

Using a generated column (auto-maintained by PostgreSQL, requires v12+) rather than a trigger. No manual sync needed — the `tsvector` updates automatically when `ocr_text` changes.

### AI provider configuration table:
```sql
CREATE TABLE IF NOT EXISTS gnucash_web_ai_config (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES gnucash_web_users(id) ON DELETE CASCADE,
    provider VARCHAR(50) NOT NULL DEFAULT 'none',
    base_url VARCHAR(500),
    api_key_encrypted TEXT,
    model VARCHAR(100),
    enabled BOOLEAN NOT NULL DEFAULT FALSE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id)
);
```

Provider values: `none` (regex only), `ollama`, `openai`, `custom` (any OpenAI-compatible endpoint).

**Note:** Anthropic's API is not OpenAI-compatible (different request/response format). Users who want to use Claude can route through an OpenAI-compatible proxy (e.g., LiteLLM) and select `custom` provider. Native Anthropic adapter is deferred.

API key encryption follows the existing SimpleFin pattern in `gnucash_web_simplefin_connections.access_url_encrypted`.

## Architecture

### Receipt Extraction Pipeline

```
Receipt Upload → OCR (existing) → Extraction → Store extracted_data
                                      │
                            ┌─────────┴──────────┐
                            │                     │
                     Regex Extractor      AI Extractor
                     (always available)   (if configured)
                            │                     │
                            └─────────┬──────────┘
                                      │
                              extracted_data JSON
                                      │
                         (stored in DB, matching computed on-demand)
```

### Extraction Module (`src/lib/receipt-extraction.ts`)

**Regex Extractor (default):**
- Amount: Match patterns like `$42.17`, `TOTAL: 42.17`, `AMOUNT DUE 42.17`, `GRAND TOTAL 42.17`, etc. Returns the largest dollar amount found (most likely the total).
- Date: Match `MM/DD/YYYY`, `YYYY-MM-DD`, `Mar 15, 2026`, `03/15/26`, etc. Returns the first date found.
- Vendor: First non-numeric, non-date line of OCR text (usually the store name at top of receipt).

**AI Extractor (pluggable, OpenAI-compatible API):**
- Single `fetch()` call to the configured `base_url + /chat/completions`
- System prompt: "Extract structured data from this receipt text. Return JSON with: amount (number), currency (string), date (YYYY-MM-DD), vendor (string)."
- Model configurable per user (e.g., `llama3.2` for Ollama, `gpt-4o-mini` for OpenAI)
- Timeout: 30 seconds. On failure: fall back to regex silently, set `extraction_method: "ai_fallback_regex"`.
- No SDK dependency — raw `fetch()` to OpenAI-compatible `/v1/chat/completions` endpoint.

**OCR pipeline integration:**

The extraction step is added to the existing `handleOcrReceipt` BullMQ job in `src/lib/queue/jobs/ocr-receipt.ts`. After OCR text is extracted and saved:

```typescript
// Existing: extract OCR text and save
await updateOcrResults(receiptId, extractedText, 'complete');

// New: run structured extraction on the OCR text
const receipt = await getReceiptById(receiptId); // has created_by
const aiConfig = await getAiConfig(receipt.created_by); // may be null
const extractedData = await extractReceiptData(extractedText, aiConfig);
await updateExtractedData(receiptId, extractedData);
```

New function `updateExtractedData(id, data)` added to `src/lib/receipts.ts`:
```sql
UPDATE gnucash_web_receipts SET extracted_data = $1, updated_at = NOW() WHERE id = $2
```

**ocr_status values** (complete enum): `pending`, `processing`, `complete`, `failed`, `enqueue_failed`. Extraction failures do NOT change `ocr_status` — the OCR succeeded, only the structured parsing failed. Extraction status is tracked via `extracted_data.extraction_method` being present (extracted) or null (not yet extracted).

### Scoring Engine (`src/lib/receipt-matching.ts`)

Given a receipt's `extracted_data`, find candidate transactions via a **single batched SQL query**:

```sql
-- Fetch candidate transactions within date window with split amounts
SELECT DISTINCT t.guid, t.description, t.post_date,
       ABS(s.value_num::decimal / s.value_denom) as amount
FROM transactions t
JOIN splits s ON s.tx_guid = t.guid
JOIN accounts a ON a.guid = s.account_guid
WHERE t.post_date BETWEEN ($receipt_date - INTERVAL '7 days') AND ($receipt_date + INTERVAL '7 days')
  AND a.guid = ANY($book_account_guids::text[])
  AND t.guid NOT IN (
    SELECT DISTINCT transaction_guid FROM gnucash_web_receipts
    WHERE transaction_guid IS NOT NULL AND book_guid = $book_guid
    AND id != $receipt_id  -- allow multiple receipts per transaction
  )
```

If receipt date is null (extraction failed), widen to 30 days from receipt upload date.

**Scoring in TypeScript** (not SQL — keeps the scoring logic testable):

1. **Amount match (weight: 0.5):**
   - Exact match (within $0.01): 1.0
   - Within 1%: 0.8 (handles rounding)
   - Within 5%: 0.5 (handles tip variations)
   - Beyond 5%: 0.0

2. **Date match (weight: 0.3):**
   - Same day: 1.0
   - ±1 day: 0.9 (posting delay)
   - ±2-3 days: 0.7
   - ±4-7 days: 0.4

3. **Vendor match (weight: 0.2):**
   - Normalize both strings: lowercase, strip numbers/punctuation, collapse whitespace
   - Exact normalized match: 1.0
   - One contains the other: 0.7 (e.g., "costco" in "costco wholesale #482")
   - Uses `fastest-levenshtein` npm package (TypeScript, no PostgreSQL extension needed): distance < 3: 0.5
   - No match: 0.0

**Combined score:** `amount_score * 0.5 + date_score * 0.3 + vendor_score * 0.2`

**Candidate filtering:**
- Exclude transactions in `extracted_data.dismissed_guids` for this receipt
- Score threshold: 0.3 minimum to appear as suggestion
- Return top 5 candidates ranked by score

### Match candidates are always computed on-demand (never cached in DB). This avoids staleness when new transactions arrive. The single batched SQL query + TypeScript scoring completes in < 100ms for typical datasets.

### Query changes in `src/lib/receipts.ts` `listReceipts()`:

```typescript
// Replace the ILIKE search condition:
if (params.search) {
  conditions.push(`r.ocr_tsvector @@ plainto_tsquery('english', $${paramIdx})`);
  values.push(params.search);
  paramIdx++;
}

// Update ORDER BY: use relevance ranking when searching, date when not
const orderBy = params.search
  ? `ts_rank(r.ocr_tsvector, plainto_tsquery('english', $${searchParamIdx})) DESC`
  : 'r.created_at DESC';
```

`plainto_tsquery` handles user input naturally — "costco milk" becomes `costco & milk`. Supports stemming, stop word removal, and ranking.

## Open Questions (Resolved)

1. **API keys:** Primary source is `AI_API_KEY` env var (like `DATABASE_URL`). DB storage via settings page is optional override, encrypted with AES-256 using `SESSION_SECRET` as key material. Decryption failures (SESSION_SECRET changed) return null with clear "re-enter key" error.
2. **Match suggestion caching:** No caching — always compute on-demand. The batched SQL + TypeScript scoring is fast enough (< 100ms), and caching introduces staleness when new transactions arrive.
3. **Re-extraction trigger:** Deferred to TODOS.md. Users get AI on new uploads immediately.

## Engineering Review Decisions (2026-03-22)

Decisions made during `/plan-eng-review`:

1. **BullMQ concurrency:** Increase OCR job concurrency from 1 to 3-5 for faster batch processing. Parallel extraction when AI provider is slow (Ollama).
2. **API key storage:** `.env` (`AI_API_KEY`) as primary source. DB table `gnucash_web_ai_config` with AES-256 encrypted key as optional per-user override. Decryption failure returns null with clear error.
3. **Dismiss persistence:** Atomic JSONB append via PostgreSQL `jsonb_set + ||` operator. No read-modify-write race condition.
4. **Test strategy:** Unit tests only (~30 cases). No integration tests requiring test DB. Extraction and scoring are pure functions — highly testable.
5. **Scope:** All three features (FTS + matching + AI settings) ship together. No deferral.

## Success Criteria

- User can upload 10 receipts in batch mode in under 30 seconds
- Inbox shows match suggestions within 2 seconds of opening
- Amount-exact + same-day matches score > 0.8 confidence
- Full-text search returns results in < 100ms with 1000+ receipts
- AI provider can be configured, tested, and used without restarting the app
- Everything works without AI configured — regex extraction is the default

## What I noticed about how you think

- You said "start with 2, but include support for AI using a pluggable model" — you're thinking in layers. Build the reliable foundation first, add the smart layer on top. That's the right instinct for open source: the baseline must work for everyone, the enhancement rewards those who invest in setup.
- You picked "all of the above" for the audience — you're not willing to pick favorites. You want to serve the tax-prep warrior and the daily tracker and the accountant. That's ambitious, but your instinct to "build for the hardest case" is right: the batch inbox workflow serves all three.
- You immediately connected line item extraction to sales tax reporting — "a report that could parse out the amount of sales tax over a year would be really cool." You're thinking about the downstream value of structured data, not just the immediate feature. That's what makes the `extracted_data` JSONB column worth designing now even though line items aren't in scope yet.
- You chose "keep it simple" for inspiration rather than chasing Expensify or Paperless-ngx. The integration with real GnuCash accounting data IS the differentiator. You trust that.
